# Decision Log

Use this file to record important project decisions so future conversations do not silently drift.

## How To Use

For each significant decision, record:

1. date
2. decision
3. why it was made
4. alternatives considered
5. what would cause the decision to change
6. status: proposed, active, replaced, or rejected

## Entries

### 2026-03-13 - Business model should be agent-first, human-approved

- Status: active
- Why: this preserves trust, pricing control, quality control, and launch safety while still getting most of the efficiency gain from automation
- Alternatives considered: fully autonomous execution from day one
- Change trigger: only revisit after multiple client projects prove the workflows are stable enough for reduced human review

### 2026-03-13 - Initial target niches should be home services and fitness in the GTA

- Status: replaced
- Why: these niches were the initial shortlist before locking the first niche
- Alternatives considered: restaurants, real estate, ecommerce
- Change trigger: superseded by the decision to start with home services first and move fitness to second priority

### 2026-03-13 - First niche is home services in the GTA

- Status: active
- Why: home services have strong local lead value, highly repeatable marketing-site structures, clear service-area SEO opportunities, and a good fit for productized website packages
- Alternatives considered: fitness studios / gyms as co-first niche
- Change trigger: revisit only if real outreach data shows materially worse demand, close rates, or delivery economics than expected

### 2026-03-13 - Initial service tiers should be defined around home-service lead generation needs

- Status: active
- Why: Launch, Growth, and Premium need to reflect actual niche needs instead of generic agency tiers, otherwise qualification, outreach, discovery, and proposals stay too abstract
- Alternatives considered: keeping broad generic tier descriptions until later
- Change trigger: revisit after early sales calls or delivery experience show repeated mismatch between package definitions and real client needs

### 2026-03-13 - Delivery stack should be Next.js + Payload CMS

- Status: active
- Why: strong fit for block-based marketing websites, reusable templates, admin customization, and repeatable delivery
- Alternatives considered: broader CMS or website-builder-first approaches
- Change trigger: revisit only if the stack meaningfully slows delivery or blocks required client features

### 2026-03-13 - Start with productized services instead of broad custom builds

- Status: active
- Why: productized tiers improve margins, reduce delivery chaos, and make agent workflows easier to standardize
- Alternatives considered: custom agency model from the start
- Change trigger: revisit after V1 is profitable and repeatable
